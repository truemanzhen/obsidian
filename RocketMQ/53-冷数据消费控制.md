# 冷数据消费控制

> 源码位置：`broker/src/main/java/org/apache/rocketmq/broker/coldctr/`

## 🏗️ 什么是冷数据控制

冷数据控制（Cold Data Control）是 5.x 引入的**消费限流**机制——当消费者读取冷数据（历史消息）时，自动限流以保护 Broker 性能。

```
场景：
├── 热数据（最近的消息）：正常消费速度
├── 冷数据（历史消息）：限制消费速度
└── 防止冷数据读取拖垮 Broker
```

## 📝 ColdDataCgCtrService

### 数据结构

```java
public class ColdDataCgCtrService extends ServiceThread {
    // 消费组 → 冷数据读取量
    private final ConcurrentHashMap<String, AccAndTimeStamp> cgColdThresholdMapRuntime;

    // 全局冷数据读取量
    private static final AtomicLong GLOBAL_ACC = new AtomicLong(0L);

    // 自适应标记
    private static final String ADAPTIVE = "||adaptive";
}

public class AccAndTimeStamp {
    private AtomicLong acc;      // 累计读取量
    private long timestamp;       // 最后更新时间
}
```

### 核心逻辑

```java
// 检查消费组是否触发冷数据限流
public boolean isColdDataCgCtr(String consumerGroup) {
    AccAndTimeStamp accAndTs = cgColdThresholdMapRuntime.get(consumerGroup);
    if (accAndTs == null) {
        return false; // 未在限流名单中
    }

    // 检查累计读取量是否超过阈值
    long threshold = brokerConfig.getColdDataCgThreshold();
    return accAndTs.getAcc().get() > threshold;
}

// 记录冷数据读取
public void addColdDataAcc(String consumerGroup, long size) {
    AccAndTimeStamp accAndTs = cgColdThresholdMapRuntime.computeIfAbsent(
        consumerGroup,
        k -> new AccAndTimeStamp(new AtomicLong(0), System.currentTimeMillis())
    );
    accAndTs.getAcc().addAndGet(size);
    accAndTs.setTimestamp(System.currentTimeMillis());
}
```

### 定时清理

```java
@Override
public void run() {
    while (!isStopped()) {
        // 周期性重置冷数据读取统计
        long now = System.currentTimeMillis();
        for (Map.Entry<String, AccAndTimeStamp> entry : cgColdThresholdMapRuntime.entrySet()) {
            AccAndTimeStamp accAndTs = entry.getValue();
            // 超过阈值时间后重置
            if (now - accAndTs.getTimestamp() > cgColdAccResideTimeoutMills) {
                cgColdThresholdMapRuntime.remove(entry.getKey());
            }
        }
        Thread.sleep(scanInterval);
    }
}
```

## 🔄 ColdDataPullRequestHoldService

当消费者触发冷数据限流时，拉取请求会被挂起。

```
消费者拉取冷数据
    ↓
检查是否触发冷数据限流
    ├── 未触发 → 正常返回
    └── 触发 → 挂起请求
        ├── 加入 ColdDataPullRequestHoldService
        ├── 等待限流解除
        └── 唤醒请求，重新拉取
```

## ⚙️ 配置项

```properties
# 启用冷数据控制
enableColdDataCgCtr=true

# 消费组冷数据读取阈值（超过后限流）
coldDataCgThreshold=104857600  # 100MB

# 冷数据读取统计重置时间
coldDataCgAccResideTimeoutMills=60000  # 60秒

# 扫描间隔
coldDataCgCtrScanInterval=10000  # 10秒
```

## 💡 设计亮点

1. **按消费组限流**：不同消费组独立控制
2. **自适应阈值**：支持 `||adaptive` 标记动态调整
3. **定时重置**：超过时间自动解除限流
4. **请求挂起**：限流时挂起请求，限流解除后自动恢复
5. **全局统计**：GLOBAL_ACC 记录全局冷数据读取量

## ⚠️ 注意事项

1. **阈值设置**：太小会误限流，太大会失去保护作用
2. **重置时间**：太短限流效果差，太长影响正常消费
3. **热冷判断**：判断消息是否为冷数据需要额外开销
4. **全量扫描**：所有拉取请求都需要检查限流状态

## 🔗 与其他模块的关系

- **PullMessageProcessor**：拉取时检查冷数据限流
- **TieredStore**：冷数据通常存储在 TieredStore 的 Cold Tier
- **BrokerConfig**：限流阈值通过 BrokerConfig 配置
- **Admin 工具**：支持动态修改冷数据控制配置
