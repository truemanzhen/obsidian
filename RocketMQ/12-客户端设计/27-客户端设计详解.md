# 客户端设计 — Producer/Consumer 内部实现

> 📄 源码路径：`client/src/main/java/org/apache/rocketmq/client/`
> 🏷️ #核心类 #架构设计 #面试高频

## 🎯 核心思想

RocketMQ 客户端 SDK 封装了**网络通信、路由管理、负载均衡、消费进度**等复杂逻辑，
用户只需要关注消息的发送和消费。

### Producer 发送流程

```java
// DefaultMQProducerImpl.sendDefaultImpl()
private SendResult sendDefaultImpl(Message msg, CommunicationMode communicationMode,
    SendCallback sendCallback, long timeout) {

    // ① 查找路由
    TopicPublishInfo topicPublishInfo = tryToFindTopicPublishInfo(msg.getTopic());

    // ② 选择 Queue
    MessageQueue mq = topicPublishInfo.selectOneMessageQueue(lastBrokerName);

    // ③ 发送消息
    sendResult = this.sendKernelImpl(msg, mq, communicationMode, sendCallback, ...);

    // ④ 失败重试
    for (int times = 1; times <= retryTimes; times++) {
        // 换一个 Broker 重试
        mq = topicPublishInfo.selectOneMessageQueue(lastBrokerName);
        sendResult = this.sendKernelImpl(msg, mq, ...);
    }
}
```

### Queue 选择策略

```java
// TopicPublishInfo.selectOneMessageQueue()
public MessageQueue selectOneMessageQueue(final String lastBrokerName) {
    if (lastBrokerName == null) {
        // 第一次：轮询选择
        return mQList.get(sendWhichQueue.incrementAndGet() % mQList.size());
    } else {
        // 重试时：避开上次失败的 Broker
        for (int i = 0; i < mQList.size(); i++) {
            int index = sendWhichQueue.incrementAndGet() % mQList.size();
            MessageQueue mq = mQList.get(index);
            if (!mq.getBrokerName().equals(lastBrokerName)) {
                return mq;
            }
        }
        // 所有 Broker 都失败过，随机选一个
        return mQList.get(sendWhichQueue.incrementAndGet() % mQList.size());
    }
}
```

> [!tip] 故障规避机制
> Producer 维护一个 `brokerFaultTable`，记录每个 Broker 的故障时间。
> 选择 Queue 时，会跳过最近故障的 Broker，避免重复发送到故障节点。

## 🔍 核心服务详解

### 1. RebalanceService — 队列分配

```java
// RebalanceService（定时任务，每 20 秒）
public class RebalanceService extends ServiceThread {
    public void run() {
        while (!this.isStopped()) {
            this.waitForRunning(waitInterval);
            mqClientFactory.doRebalance();  // 触发所有 Consumer 的 Rebalance
        }
    }
}

// RebalanceImpl.doRebalance()
public boolean doRebalance(final boolean isOrder) {
    for (Map.Entry<String, SubscriptionData> entry : subTable.entrySet()) {
        String topic = entry.getKey();
        rebalanceByTopic(topic, isOrder);  // 按 Topic 分配
    }
}

// RebalanceImpl.rebalanceByTopic()
private boolean rebalanceByTopic(final String topic, final boolean isOrder) {
    // ① 获取 Topic 的所有 Queue
    Set<MessageQueue> mqSet = topicSubscribeInfoTable.get(topic);

    // ② 获取当前 Consumer 列表
    List<String> cidAll = mQClientFactory.findConsumerIdList(topic, consumerGroup);

    // ③ 使用分配策略计算结果
    List<MessageQueue> allocateResult = strategy.allocate(consumerGroup, clientId, mqAll, cidAll);

    // ④ 更新 ProcessQueueTable
    updateProcessQueueTableInRebalance(topic, allocateResultSet, isOrder);
}
```

### 2. 分配策略

```java
// AllocateMessageQueueAveragely — 平均分配
public List<MessageQueue> allocate(String consumerGroup, String currentCID,
    List<MessageQueue> mqAll, List<String> cidAll) {

    int index = cidAll.indexOf(currentCID);  // 当前 Consumer 的序号
    int averageSize = mqAll.size() / cidAll.size();  // 每个 Consumer 分配的 Queue 数
    int startIndex = index * averageSize;  // 起始位置

    for (int i = 0; i < averageSize; i++) {
        result.add(mqAll.get(startIndex + i));
    }
}
```

```
分配策略对比：

① AllocateMessageQueueAveragely（平均分配，默认）
   Queue: [Q0, Q1, Q2, Q3, Q4, Q5]
   Consumer-0: [Q0, Q1, Q2]
   Consumer-1: [Q3, Q4, Q5]

② AllocateMessageQueueAveragelyByCircle（环形分配）
   Consumer-0: [Q0, Q2, Q4]
   Consumer-1: [Q1, Q3, Q5]

③ AllocateMessageQueueConsistentHash（一致性 Hash）
   按 Hash 值分配，Consumer 增减时最小化重分配

④ AllocateMachineRoomNearby（机房就近）
   优先分配同机房的 Queue

⑤ AllocateMessageQueueByMachineRoom（指定机房）
   只分配指定机房的 Queue
```

### 3. PullMessageService — 消息拉取

```java
// PullMessageService（拉取线程）
public class PullMessageService extends ServiceThread {
    private final LinkedBlockingQueue<MessageRequest> messageRequestQueue;

    public void run() {
        while (!this.isStopped()) {
            MessageRequest messageRequest = messageRequestQueue.take();
            if (messageRequest.getMessageRequestMode() == MessageRequestMode.POP) {
                popMessage((PopRequest) messageRequest);   // Pop 模式
            } else {
                pullMessage((PullRequest) messageRequest); // Pull 模式
            }
        }
    }
}
```

### 4. ConsumeMessageService — 消费服务

```java
// ConsumeMessageConcurrentlyService（并发消费）
public void submitConsumeRequest(List<MessageExt> msgs, ProcessQueue processQueue,
    MessageQueue messageQueue, boolean dispatchToConcurrent) {

    // 将消息分批（默认每批 1 条）
    for (int total = 0; total < msgs.size(); ) {
        List<MessageExt> msgThis = msgs.subList(total, total + consumeBatchSize);
        ConsumeRequest consumeRequest = new ConsumeRequest(msgThis, processQueue, messageQueue);
        consumeExecutor.submit(consumeRequest);  // 提交到消费线程池
        total += consumeBatchSize;
    }
}

// ConsumeRequest.run()
public void run() {
    // ① 执行用户 Listener
    status = listener.consumeMessage(Collections.unmodifiableList(msgs), context);

    // ② 处理消费结果
    processConsumeResult(status, context, this);
}

// processConsumeResult()
public void processConsumeResult(ConsumeConcurrentlyStatus status, ...) {
    switch (status) {
        case CONSUME_SUCCESS:
            // 更新消费进度
            long offset = consumeRequest.getProcessQueue().removeMessage(consumeRequest.getMsgs());
            offsetStore.updateOffset(consumeRequest.getMessageQueue(), offset, true);
            break;
        case RECONSUME_LATER:
            // 发送重试消息
            sendMessageBack(msgs, context);
            // 更新消费进度（跳过这批消息）
            offset = consumeRequest.getProcessQueue().removeMessage(consumeRequest.getMsgs());
            offsetStore.updateOffset(consumeRequest.getMessageQueue(), offset, true);
            break;
    }
}
```

### 5. ProcessQueue — 消费队列缓存

```java
public class ProcessQueue {
    // 消费中的消息（已拉取但未消费完）
    private final TreeMap<Long, MessageExt> msgTreeMap;

    // 消费中的消息数量
    private final AtomicLong msgCount = new AtomicLong();

    // 最大消费跨度（流控用）
    private volatile long queueOffsetMax = 0L;

    // 最后拉取时间
    private volatile long lastPullTimestamp = System.currentTimeMillis();

    // 最后消费时间
    private volatile long lastConsumeTimestamp = System.currentTimeMillis();
}
```

> [!important] ProcessQueue 的作用
> ProcessQueue 是 Consumer 端的**消息缓存**：
> - 已从 Broker 拉取但未消费完的消息都存在这里
> - `msgTreeMap` 按 offset 排序，保证顺序消费
> - `msgCount` 用于流控（缓存消息数过多时暂停拉取）

```java
// 广播模式：本地文件存储
public class LocalFileOffsetStore implements OffsetStore {
    private final String storePath;  // ~/.rocketmq_offsets/{clientId}/{group}/offsets.json
    private ConcurrentMap<MessageQueue, AtomicLong> offsetTable;
}

// 集群模式：远程 Broker 存储
public class RemoteBrokerOffsetStore implements OffsetStore {
    private ConcurrentMap<MessageQueue, AtomicLong> offsetTable;

    public long readOffset(MessageQueue mq, ReadOffsetType type) {
        // 从 Broker 查询消费进度
        return mQClientFactory.getMQClientAPIImpl().queryConsumerOffset(brokerAddr, requestHeader, timeout);
    }

    public void updateOffset(MessageQueue mq, long offset, boolean increaseOnly) {
        // 本地缓存，定期同步到 Broker
        AtomicLong offsetOld = offsetTable.get(mq);
        offsetOld.compareAndSet(old, offset);
    }
}
```

## 📊 Consumer 启动流程

```
DefaultMQPushConsumer.start()
    │
    ├─ ① MQClientInstance.start()
    │     ├─ MQClientFactory 启动
    │     ├─ RebalanceService 启动（每 20 秒 Rebalance）
    │     ├─ PullMessageService 启动（拉取消息）
    │     ├─ DefaultMQProducerImpl 启动（用于 sendMessageBack）
    │     └─ 定时从 NameServer 更新路由
    │
    ├─ ② 向 Broker 注册 Consumer
    │     └─ HeartbeatData 包含 ConsumerInfo
    │
    ├─ ③ 立即触发 Rebalance
    │     └─ rebalanceByTopic() → 分配 Queue
    │
    └─ ④ 开始拉取消息
          └─ PullRequest → PullMessageService → Broker
```

## 📊 消费重试与死信队列

```
消费失败（RECONSUME_LATER）：
    │
    ▼
① 发送消息到 %RETRY%{group} Topic
   │  延迟级别：10s/30s/1m/2m/3m/4m/5m/6m/7m/8m/9m/10m/20m/30m/1h/2h
   │
   ▼
② Consumer 重新消费重试消息
   │  reconsumeTimes++
   │
   ├─ 消费成功 → 正常处理
   │
   └─ 继续失败
      │
      ├─ reconsumeTimes < 16 → 继续重试
      │
      └─ reconsumeTimes >= 16 → 发送到 %DLQ%{group}（死信队列）
           │  需要人工介入处理
           └─ 死信队列消息不会被自动消费
```

## 💡 设计亮点

### 1. 推拉结合

```
Push = Pull + 长轮询 + 自动管理
  - PullMessageService 持续拉取
  - 没有消息时长轮询等待
  - 有消息时回调 MessageListener
```

### 2. 故障规避

```
Producer 维护 brokerFaultTable：
  - 记录每个 Broker 的故障时间
  - 选择 Queue 时跳过故障 Broker
  - 故障恢复后自动重新使用
```

### 3. 流控机制

```
Consumer 端多级流控：
  ① ProcessQueue.msgCount > 阈值 → 暂停拉取
  ② ProcessQueue 消费跨度 > 阈值 → 暂停拉取
  ③ Broker 返回流控 → 延迟重试
```
