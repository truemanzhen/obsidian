## 🔄 GroupTransferService（同步双写）

这是 HA 的**核心组件**——Master 收到同步写请求后，等待 Slave 确认。

### 工作流程

```
Master 收到同步写请求（SYNC_MASTER 模式）
    ↓
CommitLog.putMessage(msg)
    ├── 写入本地 CommitLog
    └── 创建 GroupCommitRequest
        ├── 设置 nextOffset（写入位置）
        └── 设置 flushOK（是否需要 Slave 确认）
    ↓
GroupTransferService.putRequest(request)
    ├── 加入 requestsWrite 队列
    └── 唤醒 GroupTransferService 线程
    ↓
GroupTransferService.run()
    ├── 等待 Slave 同步完成
    │   ├── 检查 Slave 的同步 offset
    │   └── 如果 Slave offset >= nextOffset → 确认成功
    └── 唤醒等待的请求
    ↓
CommitLog.GroupCommitRequest.wakeupCustomer(flushOK)
    ↓
返回 PutMessageResult
```

### 关键代码

```java
public class GroupTransferService extends ServiceThread {
    private volatile List<CommitLog.GroupCommitRequest> requestsWrite = new ArrayList<>();
    private volatile List<CommitLog.GroupCommitRequest> requestsRead = new ArrayList<>();

    public void putRequest(final CommitLog.GroupCommitRequest request) {
        synchronized (this.requestsWrite) {
            this.requestsWrite.add(request);
        }
        if (hasNotified.compareAndSet(false, true)) {
            waitPoint.countDown(); // 唤醒
        }
    }

    private void doWaitTransfer() {
        // 交换读写队列
        List<CommitLog.GroupCommitRequest> requests;
        synchronized (this.requestsWrite) {
            requests = this.requestsWrite;
            this.requestsWrite = this.requestsRead;
            this.requestsRead = requests;
        }

        for (CommitLog.GroupCommitRequest req : requests) {
            // 等待 Slave 同步
            boolean transferOK = false;
            for (int i = 0; i < 10 && !transferOK; i++) {
                // 检查 Slave 的同步 offset
                if (haService.getPush2SlaveMaxOffset() >= req.getNextOffset()) {
                    transferOK = true;
                    break;
                }
                Thread.sleep(10); // 等待 10ms
            }

            // 唤醒等待的 putMessage 线程
            req.wakeupCustomer(transferOK);
        }
    }
}
```

## 🔄 DefaultHAClient（Slave 同步客户端）

### 同步流程

```
Slave 启动
    ↓
DefaultHAClient.run()
    ├── 连接 Master
    ├── 发送同步请求
    │   ├── 发送 Slave 的 commitLog offset
    │   └── Master 根据 offset 发送增量数据
    ├── 接收数据
    │   ├── 读取 Master 发送的 CommitLog 数据
    │   ├── 写入本地 CommitLog
    │   └── 更新本地 offset
    └── 循环
```

### 数据传输格式

```
Master → Slave 的数据格式：
┌────────────────────────────────────────┐
│ 8 bytes │ Slave 请求的 offset          │
│ 8 bytes │ Master 当前最大 offset       │
│ 4 bytes │ 数据长度                      │
│ N bytes │ CommitLog 数据                │
└────────────────────────────────────────┘
```

## 🔄 AutoSwitchHAService（自动切换）

### 与 DefaultHAService 的区别

| 特性 | DefaultHAService | AutoSwitchHAService |
|---|---|---|
| 选主方式 | 手动 | 自动（Controller/Raft） |
| Epoch 机制 | 无 | 有（防止脑裂） |
| 角色切换 | 需要重启 | 热切换 |
| 数据一致性 | 最终一致 | 强一致 |

### 角色切换流程

```
Controller 通知角色切换
    ↓
AutoSwitchHAService.changeToMaster(epoch)
    ├── 更新本地 Epoch
    ├── 停止 HAClient（不再从 Master 同步）
    ├── 启动 HAConnection（开始接受 Slave 同步）
    ├── 更新 CommitLog 的可写状态
    └── 注册到 NameServer

AutoSwitchHAService.changeToSlave(masterAddr)
    ├── 停止 HAConnection
    ├── 启动 HAClient（开始从 Master 同步）
    ├── 更新 CommitLog 为只读
    └── 取消注册
```

## 💡 设计亮点

1. **读写队列分离**：GroupTransferService 用两个 List 交替，减少锁竞争
2. **超时机制**：同步等待有超时，避免 Slave 宕机阻塞 Master
3. **Epoch 防脑裂**：AutoSwitchHA 用 Epoch 确保只有一个 Master
4. **增量同步**：Slave 只同步增量数据，不全量复制
5. **热切换**：AutoSwitchHA 不需要重启即可切换角色

## ⚠️ 注意事项

1. **同步延迟**：Slave 网络延迟影响 Master 写入性能
2. **超时配置**：同步超时需要根据网络情况调整
3. **Epoch 文件**：需要持久化，重启时恢复
4. **Slave 离线**：Slave 长时间离线后需要全量同步
