# 长轮询机制 — 三种长轮询实现详解

> 📄 源码路径：`broker/src/main/java/org/apache/rocketmq/broker/longpolling/`
> 🏷️ #核心类 #设计模式 #面试高频

## 🎯 核心思想

长轮询 = **挂起请求 + 消息到达通知 + 超时返回**。
用"拉"的方式模拟"推"的效果，兼顾实时性和资源效率。

## 📐 1. PullRequestHoldService — Pull 模式长轮询

### 数据结构

```java
public class PullRequestHoldService extends ServiceThread {
    // 挂起的请求表：topic@queueId → 请求列表
    protected ConcurrentMap<String, ManyPullRequest> pullRequestTable;
}
```

### 挂起请求

```java
public void suspendPullRequest(String topic, int queueId, PullRequest pullRequest) {
    String key = buildKey(topic, queueId);  // "topic@queueId"
    ManyPullRequest mpr = pullRequestTable.get(key);
    mpr.addPullRequest(pullRequest);
}
```

### 唤醒机制

```java
// 定时检查（每 5 秒）
protected void checkHoldRequest() {
    for (String key : pullRequestTable.keySet()) {
        String topic = kArray[0];
        int queueId = Integer.parseInt(kArray[1]);
        long offset = messageStore.getMaxOffsetInQueue(topic, queueId);
        notifyMessageArriving(topic, queueId, offset);
    }
}

// 消息到达通知
public void notifyMessageArriving(String topic, int queueId, long maxOffset,
    Long tagsCode, long msgStoreTime, byte[] filterBitMap, Map<String, String> properties) {

    List<PullRequest> requestList = mpr.cloneListAndClear();
    for (PullRequest request : requestList) {
        if (newestOffset > request.getPullFromThisOffset()) {
            // 有新消息 → 唤醒请求
            pullMessageProcessor.executeRequestWhenWakeup(request);
        } else if (System.currentTimeMillis() >= request.getSuspendTimestamp() + request.getTimeoutMillis()) {
            // 超时 → 返回空
            pullMessageProcessor.executeRequestWhenWakeup(request);
        } else {
            // 继续等待
            replayList.add(request);
        }
    }
}
```

### 触发来源

```
PullRequestHoldService 的通知来源：

① 定时检查（每 5 秒）
   → checkHoldRequest() 遍历所有挂起的请求

② ReputMessageService 分发
   → NotifyMessageArrivingListener.arriving()
   → pullRequestHoldService.notifyMessageArriving()

③ Consumer ACK（顺序消费场景）
   → ack 后唤醒等待的请求
```

## 📐 2. PopLongPollingService — Pop 模式长轮询

### 数据结构

```java
public class PopLongPollingService extends ServiceThread {
    // Caffeine 缓存：topic@cid@queueId → 请求集合
    private final Cache<String, ConcurrentSkipListSet<PopRequest>> pollingMap;

    // topic → cid 映射（用于消息到达时快速找到等待的 Consumer）
    private final Cache<String, ConcurrentHashMap<String, Byte>> topicCidMap;
}
```

### 挂起请求

```java
public PollingResult polling(ChannelHandlerContext ctx, RemotingCommand request,
    PollingHeader pollingHeader, SubscriptionData subscriptionData, MessageFilter messageFilter) {

    // 构建 PopRequest
    PopRequest popRequest = new PopRequest(ctx, request, pollingHeader, subscriptionData, messageFilter);

    // 添加到 pollingMap
    String key = KeyBuilder.buildPollingKey(topic, cid, queueId);
    pollingMap.getIfPresent(key).add(popRequest);

    return PollingResult.POLLING_SUC;
}
```

### 唤醒机制

```java
// 定时检查（每 20ms）
public void run() {
    while (!this.stopped) {
        this.waitForRunning(20);

        for (Map.Entry<String, ConcurrentSkipListSet<PopRequest>> entry : pollingMap.asMap().entrySet()) {
            PopRequest first = popQ.pollFirst();
            if (first == null) break;

            if (!first.isTimeout()) {
                // 未超时，放回队列
                popQ.add(first);
                break;
            }

            // 超时，唤醒
            wakeUp(first);
        }
    }
}

// 消息到达通知
public boolean notifyMessageArriving(String topic, int queueId, String cid, boolean force,
    Long tagsCode, long msgStoreTime, byte[] filterBitMap, Map<String, String> properties) {

    // 找到挂起的请求
    ConcurrentSkipListSet<PopRequest> remotingCommands =
        pollingMap.getIfPresent(KeyBuilder.buildPollingKey(topic, cid, queueId));

    PopRequest popRequest = pollRemotingCommands(remotingCommands);

    // 消息过滤匹配
    if (!force && popRequest.getMessageFilter() != null) {
        boolean match = popRequest.getMessageFilter().isMatchedByConsumeQueue(tagsCode, ...);
        if (!match) {
            // 不匹配，放回队列
            remotingCommands.add(popRequest);
            return false;
        }
    }

    // 唤醒
    return wakeUp(popRequest);
}
```

### 唤醒实现

```java
public boolean wakeUp(PopRequest request) {
    // 重新执行 Pop 请求
    Runnable run = () -> {
        RemotingCommand response = processor.processRequest(request.getCtx(), request.getRemotingCommand());
        NettyRemotingAbstract.writeResponse(request.getChannel(), request.getRemotingCommand(), response, ...);
    };

    // 提交到线程池执行
    brokerController.getPullMessageExecutor().submit(new RequestTask(run, ...));
    return true;
}
```

## 📐 3. NotifyMessageArrivingListener — 消息到达的统一入口

```java
public class NotifyMessageArrivingListener implements MessageArrivingListener {
    private final PullRequestHoldService pullRequestHoldService;
    private final PopMessageProcessor popMessageProcessor;
    private final NotificationProcessor notificationProcessor;
    private final LiteEventDispatcher liteEventDispatcher;

    @Override
    public void arriving(String topic, int queueId, long logicOffset, long tagsCode,
        long msgStoreTime, byte[] filterBitMap, Map<String, String> properties) {

        // 通知 Pull 长轮询
        pullRequestHoldService.notifyMessageArriving(topic, queueId, logicOffset, tagsCode, ...);

        // 通知 Pop 长轮询
        popMessageProcessor.notifyMessageArriving(topic, queueId, logicOffset, tagsCode, ...);

        // 通知 Notification（Lite Push）
        notificationProcessor.notifyMessageArriving(topic, queueId, logicOffset, tagsCode, ...);
    }
}
```

> [!important] 统一通知入口
> `NotifyMessageArrivingListener` 是消息到达的**统一通知入口**，
> 由 `ReputMessageService` 在分发消息时调用。
> 它同时通知三种长轮询服务，确保所有等待的请求都能被及时唤醒。

## 📐 长轮询完整流程图

```
Consumer 发来 Pull/Pop 请求，没有消息：

① Broker 将请求挂起
   │
   ├─ Pull: pullRequestTable["topic@queueId"].add(pullRequest)
   └─ Pop:  pollingMap["topic@cid@queueId"].add(popRequest)

② ReputMessageService 分发新消息到 ConsumeQueue
   │
   ▼
③ NotifyMessageArrivingListener.arriving()
   │
   ├─ pullRequestHoldService.notifyMessageArriving()
   │     → 遍历 pullRequestTable["topic@queueId"]
   │     → 检查：有新消息？过滤匹配？超时？
   │     → 唤醒：executeRequestWhenWakeup()
   │
   ├─ popMessageProcessor.notifyMessageArriving()
   │     → 遍历 pollingMap["topic@cid@queueId"]
   │     → 检查：过滤匹配？
   │     → 唤醒：wakeUp()
   │
   └─ notificationProcessor.notifyMessageArriving()
         → 通知 Lite Push Consumer

④ 唤醒的请求重新执行
   │
   ├─ Pull: pullMessageProcessor.processRequest() → 返回消息
   └─ Pop:  popMessageProcessor.processRequest() → 返回消息

⑤ Consumer 收到消息 ✓
```

## 📐 超时处理

```
长轮询超时时间计算：

Pull 模式：
  timeout = requestHeader.getSuspendTimeoutMillis()  // 客户端设置
  默认：15 秒（可配置）

Pop 模式：
  pollingTime = request.getLongPollingTimeout()
  限制在 [minPollingTime, maxPollingTime] 范围内
  检查 gRPC deadline 是否足够

超时后：
  → 返回空结果
  → Consumer 重新发起 Pull/Pop 请求
```

## 💡 设计亮点

### 1. 消息过滤在 Broker 端

```
长轮询唤醒时，Broker 端先做消息过滤：
  if (popRequest.getMessageFilter().isMatchedByConsumeQueue(tagsCode, ...)) {
      wakeUp(popRequest);  // 匹配才唤醒
  } else {
      remotingCommands.add(popRequest);  // 不匹配，继续等待
  }

好处：减少无效唤醒，节省网络和 CPU
```

### 2. Caffeine 缓存（Pop 模式）

```java
// 自动过期清理，避免内存泄漏
pollingMap = Caffeine.newBuilder()
    .maximumSize(popPollingMapSize)           // 最大条目数
    .expireAfterAccess(expireTimeSeconds)     // 访问后过期时间
    .build();
```

### 3. 请求合并（ManyPullRequest）

```java
// 同一个 topic@queueId 的多个请求合并管理
public class ManyPullRequest {
    private final ArrayList<PullRequest> pullRequestList = new ArrayList<>();

    public synchronized void addPullRequest(PullRequest pullRequest) {
        pullRequestList.add(pullRequest);
    }

    public synchronized List<PullRequest> cloneListAndClear() {
        List<PullRequest> result = new ArrayList<>(pullRequestList);
        pullRequestList.clear();
        return result;
    }
}
```

## 📊 长轮询 vs 短轮询 vs 推送

| 维度 | 短轮询 | 长轮询 | 推送（Server Push） |
|------|--------|--------|-------------------|
| 实现 | 定时 Pull | Pull + 挂起 | Broker 主动发 |
| 实时性 | 差（取决于间隔） | 好（毫秒级） | 最好 |
| 资源消耗 | 高（频繁请求） | 低（挂起等待） | 低 |
| 网络开销 | 大（空轮询） | 小 | 最小 |
| 实现复杂度 | 低 | 中 | 高 |
| RocketMQ | 不支持 | ✅ 支持 | 不支持（5.x 用 Pop 代替） |

## 💡 学习收获

1. **长轮询本质**：挂起请求 + 消息到达通知 + 超时返回
2. **三种实现**：Pull、Pop、Lite Push 各有对应的长轮询服务
3. **统一通知入口**：NotifyMessageArrivingListener 同时通知三种服务
4. **Broker 端过滤**：唤醒前先做过滤，减少无效唤醒
5. **Caffeine 缓存**：Pop 模式用 Caffeine 管理挂起请求，自动过期清理
