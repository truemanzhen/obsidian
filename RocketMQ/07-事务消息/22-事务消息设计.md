# 事务消息 — 两阶段提交 + 回查机制

> 📄 源码路径：`broker/src/main/java/org/apache/rocketmq/broker/transaction/`
> 🏷️ #核心类 #设计模式 #面试高频

## 🎯 核心思想

事务消息保证**本地事务和消息发送的原子性**：
- 本地事务执行成功 → 消息提交（对 Consumer 可见）
- 本地事务执行失败 → 消息回滚（删除）
- 本地事务不确定 → Broker 回查 Producer

## 📐 两阶段提交流程

```
Producer                      Broker
  │                              │
  │ ① 发送半消息 (Half Message)   │
  │─────────────────────────────→│
  │                              │ 写入 %RMQ_SYS_TRANS_HALF_TOPIC%
  │  半消息写入成功               │ (对 Consumer 不可见)
  │←─────────────────────────────│
  │                              │
  │ ② 执行本地事务               │
  │  (数据库操作等)               │
  │                              │
  │ ③ 提交/回滚                  │
  │  COMMIT 或 ROLLBACK          │
  │─────────────────────────────→│
  │                              │
  │  COMMIT:                     │
  │    恢复原始 Topic             │
  │    写入 CommitLog             │
  │    删除半消息                 │
  │    → Consumer 可见 ✓          │
  │                              │
  │  ROLLBACK:                   │
  │    删除半消息                 │
  │    → 消息消失 ✓               │
  │                              │
```

## 📐 事务消息存储结构

```
┌──────────────────────────────────────────────────────────────┐
│ 事务消息涉及的 Topic                                          │
│                                                              │
│ %RMQ_SYS_TRANS_HALF_TOPIC%    ← 半消息存储（prepare）         │
│ %RMQ_SYS_TRANS_OP_HALF_TOPIC% ← 半消息操作记录（commit/rollback）│
│ Original Topic                ← 提交后的消息存储               │
└──────────────────────────────────────────────────────────────┘

存储流程：
① Half Message → %RMQ_SYS_TRANS_HALF_TOPIC% (ConsumeQueue)
② Op Message   → %RMQ_SYS_TRANS_OP_HALF_TOPIC% (标记 prepare 状态)
③ Commit 后    → Original Topic (对 Consumer 可见)
④ 删除 Half    → 在 Op 中标记已处理
```

```java
// TransactionalMessageBridge
public CompletableFuture<PutMessageResult> asyncPutHalfMessage(MessageExtBrokerInner messageInner) {
    // 修改 Topic 为半消息 Topic
    messageInner.setTopic(TransactionalMessageUtil.buildHalfTopic());
    // 修改 QueueId 为 0
    messageInner.setQueueId(0);

    // 写入 CommitLog
    return this.brokerController.getMessageStore().asyncPutMessage(messageInner);
}
```

### 3. 事务回查 — TransactionalMessageCheckService

```java
// TransactionalMessageCheckService（后台线程）
public class TransactionalMessageCheckService extends ServiceThread {
    public void run() {
        while (!this.isStopped()) {
            this.waitForRunning(checkInterval);  // 默认 60 秒

            // ① 扫描超时的半消息
            // ② 对每条超时半消息，调用 Producer 的回查接口
            // ③ 根据回查结果提交或回滚
        }
    }
}
```

> [!important] 回查机制
> 当 Broker 未收到 Producer 的 COMMIT/ROLLBACK 时（如网络中断、Producer 宕机）：
> 1. Broker 定期扫描超时的半消息（默认 60 秒一次）
> 2. 向 Producer 发送回查请求（checkTransactionState）
> 3. Producer 执行本地事务检查，返回 COMMIT 或 ROLLBACK
> 4. Broker 根据结果处理半消息

## 📐 事务消息完整生命周期

```
时间线 ──────────────────────────────────────────────────────────→

① Producer: sendMessage(halfMessage)
   │
   ▼
② Broker: 写入 %RMQ_SYS_TRANS_HALF_TOPIC%
   │  消息带标记：PROPERTY_TRANSACTION_PREPARED = true
   │
   ▼
③ Producer: 执行本地事务（DB 操作）
   │
   ├─ 成功 → endTransaction(COMMIT)
   │           │
   │           ▼
   │         Broker: 恢复原始 Topic → 写入 CommitLog → 删除半消息
   │           │
   │           ▼
   │         Consumer 消费到消息 ✓
   │
   ├─ 失败 → endTransaction(ROLLBACK)
   │           │
   │           ▼
   │         Broker: 删除半消息 → 消息消失 ✓
   │
   └─ 不确定（Producer 宕机等）
              │
              ▼
         Broker: TransactionalMessageCheckService 扫描超时半消息
              │
              ▼
         Broker: 调用 Producer 回查接口
              │
              ├─ Producer 返回 COMMIT → 同上
              └─ Producer 返回 ROLLBACK → 同上
```

## 📐 事务超时保护 — checkImmunityTime

```java
// Producer 可以设置事务保护时间
// 在保护时间内，Broker 不会回查
msg.setTransactionId(...);
msg.putUserProperty(MessageConst.PROPERTY_CHECK_IMMUNITY_TIME_IN_SECONDS, "60");

// EndTransactionProcessor 中检查
if (rejectCommitOrRollback(requestHeader, prepareMessage)) {
    response.setCode(ResponseCode.ILLEGAL_OPERATION);
    // 超过保护时间，拒绝 commit/rollback，等待回查
}
```

> [!tip] checkImmunityTime 的作用
> 防止 Producer 在本地事务还未完成时，就收到 Broker 的回查请求。
> 设置保护时间后，Broker 在此时间内不会回查。

## 💡 设计亮点

### 1. 半消息对 Consumer 不可见

```
半消息写入 %RMQ_SYS_TRANS_HALF_TOPIC%（内部 Topic）
Consumer 订阅的是原始 Topic，看不到半消息
只有 COMMIT 后，消息才会被"搬"到原始 Topic
```

### 2. Op Message 记录状态

```
每条半消息都有对应的 Op Message（操作记录）：
  prepare → Op: PREPARED
  commit  → Op: COMMIT
  rollback → Op: ROLLBACK

扫描时：只有 PREPARED 且超时的消息才需要回查
```

### 3. 事务消息不支持延迟和批量

```java
// 事务消息的限制
if (MessageSysFlag.TRANSACTION_PREPARED_TYPE == tranType) {
    // 不支持延迟
    // 不支持批量
    // 只支持单条发送
}
```

## 📊 事务消息 vs 普通消息

| 维度 | 普通消息 | 事务消息 |
|------|---------|---------|
| 写入 | 直接写入目标 Topic | 先写半消息 Topic |
| 可见性 | 立即可见 | COMMIT 后才可见 |
| 回滚 | 不支持 | 支持 ROLLBACK |
| 回查 | 不支持 | 支持超时回查 |
| 性能 | 高 | 较低（两次写入） |

## 💡 学习收获

1. **两阶段提交**：半消息 → 本地事务 → COMMIT/ROLLBACK
2. **回查机制**：超时扫描 + Producer 回查，保证最终一致性
3. **半消息隔离**：用内部 Topic 隔离，Consumer 不可见
4. **Op Message**：记录半消息状态，便于扫描和回查
5. **checkImmunityTime**：保护本地事务未完成时不被回查
