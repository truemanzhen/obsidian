# 存储架构总览 — 三大文件设计

> 📄 源码路径：`store/src/main/java/org/apache/rocketmq/store/`
> 🏷️ #核心类 #架构设计 #面试高频

## 🎯 核心思想

RocketMQ 的存储采用 **顺序写 + 零拷贝** 的经典设计，核心由三个文件组成：

```
┌─────────────────────────────────────────────────────────────────┐
│                        存储架构全景                               │
│                                                                 │
│  Producer                                                       │
│     │                                                           │
│     ▼                                                           │
│  ┌──────────────────────────────────────────────────────────┐   │
│  │                    CommitLog                              │   │
│  │  ┌────────┬────────┬────────┬────────┬────────┬────────┐ │   │
│  │  │ Msg 0  │ Msg 1  │ Msg 2  │ Msg 3  │ Msg 4  │ Msg 5  │ │   │
│  │  └────────┴────────┴────────┴────────┴────────┴────────┘ │   │
│  │  所有 Topic 的消息混存，按写入顺序追加（顺序写）           │   │
│  └────────────────────────┬─────────────────────────────────┘   │
│                           │ ReputMessageService                 │
│                           │ (异步分发)                           │
│          ┌────────────────┼────────────────┐                    │
│          ▼                ▼                ▼                    │
│  ┌──────────────┐ ┌──────────────┐ ┌──────────────┐            │
│  │ ConsumeQueue  │ │ ConsumeQueue  │ │  IndexFile   │            │
│  │ TopicA-Q0    │ │ TopicA-Q1    │ │ (按Key索引)   │            │
│  │ ┌──┬──┬──┐  │ │ ┌──┬──┬──┐  │ │              │            │
│  │ │20│20│20│  │ │ │20│20│20│  │ │              │            │
│  │ └──┴──┴──┘  │ │ └──┴──┴──┘  │ │              │            │
│  │ 每条20字节   │ │ 每条20字节   │ │              │            │
│  │ 定长索引     │ │ 定长索引     │ │              │            │
│  └──────────────┘ └──────────────┘ └──────────────┘            │
│                                                                 │
│  Consumer ← 通过 ConsumeQueue 定位消息 → 从 CommitLog 读取消息   │
└─────────────────────────────────────────────────────────────────┘
```

## 📐 三大文件职责

| 文件 | 职责 | 数据格式 | 特点 |
|------|------|----------|------|
| **CommitLog** | 存储所有消息的完整内容 | 变长消息 | 顺序写，所有 Topic 混存 |
| **ConsumeQueue** | 消息的消费索引 | 定长 20 字节/条 | 按 Topic-Queue 组织 |
| **IndexFile** | 按 Key 查询的索引 | Hash 索引 | 支持按 MessageKey/UniqueKey 查询 |

## 🔍 为什么这样设计？

### 1. CommitLog：所有消息混存

```
传统设计（每个 Queue 一个文件）：
  TopicA-Q0: [Msg0][Msg1][Msg2]
  TopicA-Q1: [Msg3][Msg4]
  TopicB-Q0: [Msg5][Msg6]
  → 随机写！磁盘寻址慢

RocketMQ 设计（所有消息写同一个文件）：
  CommitLog: [Msg0][Msg3][Msg1][Msg5][Msg2][Msg4][Msg6]
  → 顺序写！磁盘吞吐高
```

> [!important] 顺序写 vs 随机写
> - 机械硬盘顺序写：~200MB/s，随机写：~1MB/s（差 200 倍！）
> - SSD 顺序写：~500MB/s，随机写：~100MB/s（差 5 倍）
> - RocketMQ 选择顺序写 CommitLog，牺牲读取便利性换取写入性能

### 2. ConsumeQueue：轻量级索引

```
ConsumeQueue 每条记录只有 20 字节：
┌───────────────────────┬─────────────┬───────────────────┐
│ CommitLog Offset (8B) │ Size (4B)   │ Tag HashCode (8B) │
└───────────────────────┴─────────────┴───────────────────┘

好处：
- 定长记录 → 可通过 offset 直接计算位置（O(1)）
- 体积小 → 全量加载到内存也才几十 MB
- 只存物理 offset → 索引和数据分离
```

### 3. 读取流程：两次查找

```
Consumer 请求 TopicA-Q0 的第 N 条消息：

① 第一次查找：ConsumeQueue
   offset=N → 计算文件位置 → 读取 20 字节索引
   得到：CommitLogOffset=1024, Size=256

② 第二次查找：CommitLog
   offset=1024 → 读取 256 字节 → 得到完整消息

看似两次查找，但 ConsumeQueue 通常在 PageCache 中，非常快。
```

## 📊 文件大小配置

| 配置项 | 默认值 | 说明 |
|--------|--------|------|
| `mappedFileSizeCommitLog` | 1GB | CommitLog 单文件大小 |
| `mappedFileSizeConsumeQueue` | 30MB | ConsumeQueue 单文件大小 |
| `mappedFileSizeIndex` | 400MB | IndexFile 单文件大小 |
| `fileReservedTime` | 72h | 文件保留时间 |

> [!tip] 为什么 CommitLog 是 1GB？
> - 太小：文件数量多，管理开销大
> - 太大：MappedByteBuffer 映射大文件可能 OOM
> - 1GB 是 MappedByteBuffer 在 32 位系统上的安全上限（64 位系统可以更大）

## 💡 学习收获

1. **顺序写 + 索引**：牺牲读取便利性换取写入吞吐，是经典的 LSM-tree 思想
2. **数据与索引分离**：CommitLog 存数据，ConsumeQueue 存索引，各司其职
3. **定长 vs 变长**：ConsumeQueue 用定长记录实现 O(1) 查找
4. **PageCache 加速**：ConsumeQueue 体积小，可以全量缓存在 OS PageCache 中
