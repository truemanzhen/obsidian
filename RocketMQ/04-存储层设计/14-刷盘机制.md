# 刷盘机制 — 消息持久化的保证

> 📄 源码路径：`store/src/main/java/org/apache/rocketmq/store/CommitLog.java`（内部类）
> 🏷️ #核心类 #面试高频

## 🎯 核心问题

消息写入 MappedByteBuffer 后，数据在 **OS PageCache** 中，并不在磁盘上。
如果此时机器宕机，数据会丢失。刷盘就是把 PageCache 中的数据**强制写入磁盘**。

## 📐 两种刷盘模式

```
┌───────────────────────────────────────────────────────────────┐
│                     同步刷盘 (SYNC_FLUSH)                      │
│                                                               │
│  Producer → CommitLog → 等待刷盘完成 → 返回成功                │
│                          ↑                                    │
│                    GroupCommitService                          │
│                    (阻塞等待 flush 完成)                       │
│                                                               │
│  特点：数据零丢失，但吞吐量低                                   │
│  适用：金融、支付等对数据可靠性要求极高的场景                     │
└───────────────────────────────────────────────────────────────┘

┌───────────────────────────────────────────────────────────────┐
│                     异步刷盘 (ASYNC_FLUSH)                     │
│                                                               │
│  Producer → CommitLog → 立即返回成功                           │
│                          ↓                                    │
│              FlushRealTimeService（后台线程）                  │
│              定期 flush（默认 500ms）                          │
│                                                               │
│  特点：吞吐量高，但宕机可能丢失最近几条消息                      │
│  适用：大多数业务场景                                           │
└───────────────────────────────────────────────────────────────┘
```

## 🔍 刷盘服务详解

### 1. GroupCommitService — 同步刷盘

```java
class GroupCommitService extends FlushCommitLogService {
    // 请求队列
    private volatile List<GroupCommitRequest> requestsWrite = new ArrayList<>();
    private volatile List<List<GroupCommitRequest>> flushDiskTimes = new ArrayList<>();

    public synchronized void putRequest(final GroupCommitRequest request) {
        // 交换读写队列（CopyOnWrite 思想）
        synchronized (this) {
            this.requestsWrite.add(request);
            if (!hasNotified) {
                hasNotified = true;
                this.notify();  // 唤醒刷盘线程
            }
        }
    }

    public void run() {
        while (!this.isStopped()) {
            this.waitForRunning(10);  // 等待被唤醒

            // 交换读写队列
            this.doSwap();

            // 执行刷盘
            this.flushDiskRequests();

            // 通知等待的生产者：刷盘完成
            this.notifyRequests();
        }
    }

    private void flushDiskRequests() {
        for (GroupCommitRequest req : this.requestsRead) {
            // 执行刷盘
            CommitLog.this.mappedFileQueue.flush(0);  // flushLeastPages=0 表示强制刷
            // 设置刷盘结果
            req.wakeupCustomer(PutMessageStatus.PUT_OK);
        }
    }
}
```

> [!important] 同步刷盘的阻塞机制
> ```java
> // 生产者端
> GroupCommitRequest request = new GroupCommitRequest(offset, timeout);
> service.putRequest(request);  // 提交刷盘请求
> CompletableFuture<PutMessageStatus> flushOkFuture = request.future();
> flushStatus = flushOkFuture.get(timeout, TimeUnit.MILLISECONDS);  // 阻塞等待
> ```
> 生产者线程会阻塞，直到 GroupCommitService 完成刷盘并唤醒。

### 2. FlushRealTimeService — 异步刷盘

```java
class FlushRealTimeService extends FlushCommitLogService {
    public void run() {
        while (!this.isStopped()) {
            boolean flushCommitLogTimed = this.isCommitLogTimed();

            // 等待：定时（默认500ms）或被唤醒
            this.waitForRunning(flushCommitLogTimed ? 1000 : flushIntervalCommitLog);

            // 执行刷盘
            CommitLog.this.mappedFileQueue.flush(flushCommitLogLeastPages);

            // 记录刷盘时间（监控用）
        }
    }
}
```

> [!note] 异步刷盘的两个参数
> | 参数 | 含义 | 默认值 |
> |------|------|--------|
> | `flushIntervalCommitLog` | 刷盘间隔 | 500ms |
> | `flushCommitLogLeastPages` | 最少积攒页数才刷 | 4 页（16KB） |
>
> 两个条件满足其一就刷盘：**定时触发** 或 **积攒够数据**。

### 3. CommitRealTimeService — TransientStorePool 模式

```java
class CommitRealTimeService extends FlushCommitLogService {
    public void run() {
        while (!this.isStopped()) {
            // 等待
            this.waitForRunning(intervalCommit);

            // commit：堆外内存 → FileChannel（提交到 PageCache）
            CommitLog.this.mappedFileQueue.commit(commitLeastPages);
        }
    }
}
```

> [!tip] 两次提交模式（TransientStorePool）
> ```
> 堆外内存池 ──commit()──→ FileChannel ──flush()──→ 磁盘
>              ↑                        ↑
>      CommitRealTimeService    FlushRealTimeService
> ```
> - `commit()`：堆外内存 → PageCache（快，因为是内存拷贝）
> - `flush()`：PageCache → 磁盘（慢，因为是磁盘 I/O）
> - 两次操作分离，可以更灵活地控制时机

## 📐 MappedFileQueue.flush() 实现

```java
public boolean flush(final int flushLeastPages) {
    // ① 找到需要刷盘的文件
    MappedFile mappedFile = this.findMappedFileByOffset(
        this.getFlushedWhere(), this.getFlushedWhere() == 0);

    if (mappedFile != null) {
        // ② 执行文件级刷盘
        int offset = mappedFile.flush(flushLeastPages);

        // ③ 更新全局刷盘位置
        long where = mappedFile.getFileFromOffset() + offset;
        this.setFlushedWhere(where);
    }
}
```

### MappedFile.flush() — 文件级刷盘

```java
// DefaultMappedFile
public int flush(final int flushLeastPages) {
    // 计算需要刷盘的数据量
    int dataRemaining = this.wrotePosition - this.flushedPosition;

    // 如果积攒的数据不够 flushLeastPages 页，跳过
    if (flushLeastPages > 0) {
        if ((dataRemaining / OS_PAGE_SIZE) < flushLeastPages) {
            return this.flushedPosition;  // 不刷
        }
    }

    // 执行刷盘
    if (this.fileChannel != null) {
        this.fileChannel.force(false);  // FileChannel 模式
    } else {
        this.mappedByteBuffer.force();  // MappedByteBuffer 模式
    }

    this.flushedPosition = this.wrotePosition;
    return this.flushedPosition;
}
```

## 📊 刷盘类型对比

| | 同步刷盘 | 异步刷盘 | TransientStorePool |
|---|---------|---------|-------------------|
| 配置 | `SYNC_FLUSH` | `ASYNC_FLUSH` | `ASYNC_FLUSH` + 堆外池 |
| 数据安全 | 零丢失 | 最多丢 500ms 数据 | 最多丢 commit+flush 间隔 |
| 吞吐量 | 低 | 高 | 最高 |
| 延迟 | 高（等刷盘） | 低（立即返回） | 低 |
| 适用场景 | 金融 | 通用 | 高吞吐 |

## 💡 设计亮点

### 1. CopyOnWrite 队列交换

```java
// GroupCommitService 用两个队列交替使用
private volatile List<GroupCommitRequest> requestsWrite = new ArrayList<>();
private volatile List<GroupCommitRequest> requestsRead = new ArrayList<>();

// 写入时加锁
synchronized (this) {
    this.requestsWrite.add(request);
}

// 刷盘时交换
private void doSwap() {
    List<GroupCommitRequest> tmp = this.requestsRead;
    this.requestsRead = this.requestsWrite;
    this.requestsWrite = tmp;
}
```

> [!tip] 生产者-消费者无锁化
> 写入和刷盘操作在不同队列上，交换队列时才需要同步。
> 这比用 ConcurrentLinkedQueue 更高效。

### 2. 定时 + 定量双触发

```
异步刷盘的触发条件（满足其一）：
① 定时：500ms 到了 → 刷盘
② 定量：积攒 ≥ 4 页（16KB）→ 刷盘

好处：
- 低流量时：定时触发，保证延迟 ≤ 500ms
- 高流量时：定量触发，减少刷盘次数
```

### 3. waitForRunning 机制

```java
protected void waitForRunning(long interval) {
    if (hasNotified.compareAndSet(true, false)) {
        this.waitForRunning(interval);  // 被唤醒或超时
    } else {
        // 没有通知，阻塞等待
        this.waitForRunning(interval);
    }
}
```

> [!note] 唤醒机制
> 生产者写入后可以调用 `flushCommitLogService.wakeup()` 立即触发刷盘，
> 而不是等定时器超时。这对低延迟场景很有用。

## 💡 学习收获

1. **PageCache 是关键**：写入 PageCache 就算"写入成功"，刷盘是持久化保障
2. **同步 vs 异步**：数据安全 vs 吞吐量的经典权衡
3. **两次提交**：TransientStorePool 将 commit 和 flush 分离，更灵活
4. **CopyOnWrite 队列交换**：无锁化的生产者-消费者模式
5. **定时+定量双触发**：兼顾延迟和吞吐
