# HA 主从同步 — 数据复制与故障转移

> 📄 源码路径：`store/src/main/java/org/apache/rocketmq/store/ha/`
> 🏷️ #核心类 #架构设计 #面试高频

## 🎯 核心思想

HA（High Availability）保证 Broker 主从之间的**数据同步**和**故障自动转移**。

5.5.0 支持三种 HA 模式：
1. **DefaultHA**：经典主从同步（4.x）
2. **AutoSwitchHA**：自动切换主从（5.x）
3. **Controller 模式**：基于 DLedger 的选主（5.x）

## 📐 1. DefaultHA — 经典主从同步

### 架构

```
┌──────────────────────────────────────────────────────────────┐
│                     DefaultHA 架构                            │
│                                                              │
│  ┌──────────────┐          ┌──────────────┐                 │
│  │   Master      │          │   Slave       │                 │
│  │              │          │              │                 │
│  │ CommitLog    │  ──TCP──→│ CommitLog    │                 │
│  │              │  同步     │              │                 │
│  │ HAConnection │          │ HAClient     │                 │
│  │ (服务端)     │          │ (客户端)     │                 │
│  └──────────────┘          └──────────────┘                 │
│                                                              │
│  Master:                                                    │
│    AcceptSocketService → 监听 Slave 连接                     │
│    HAConnection → 处理单个 Slave 的数据同步                   │
│    GroupTransferService → 等待同步完成                        │
│                                                              │
│  Slave:                                                     │
│    HAClient → 连接 Master，拉取数据                          │
└──────────────────────────────────────────────────────────────┘
```

### 同步流程

```
Master 端（HAConnection）：
  ① Slave 连接后，读取 Slave 上报的 maxOffset
  ② 从 CommitLog 的 maxOffset 开始，向 Slave 推送数据
  ③ 推送完成，通知 GroupTransferService

Slave 端（HAClient）：
  ① 连接 Master
  ② 上报自己的 maxOffset（8 字节）
  ③ 接收 Master 推送的数据
  ④ 写入本地 CommitLog
  ⑤ 更新 maxOffset，继续上报
```

### 同步/异步复制

```java
// 配置项：brokerRole
// SYNC_MASTER：同步复制（Master 等待 Slave 写入成功）
// ASYNC_MASTER：异步复制（Master 不等待 Slave）

// 同步复制时，使用 GroupTransferService
public class GroupTransferService extends ServiceThread {
    public void putRequest(CommitLog.GroupCommitRequest request) {
        this.requestsWrite.add(request);
    }

    // 检查 Slave 是否追上
    public void doWaitTransfer() {
        for (GroupCommitRequest req : requestsRead) {
            long untilWhere = req.getNextOffset();
            // 等待 Slave 追上 untilWhere
            if (push2SlaveMaxOffset.get() >= untilWhere) {
                req.wakeupCustomer(PutMessageStatus.PUT_OK);
            }
        }
    }
}
```

> [!important] 同步 vs 异步复制
> | | 同步复制 (SYNC_MASTER) | 异步复制 (ASYNC_MASTER) |
> |---|---|---|
> | 写入延迟 | 高（等 Slave） | 低（不等） |
> | 数据安全 | 零丢失 | 可能丢少量 |
> | 吞吐量 | 低 | 高 |
> | 适用 | 金融 | 通用 |

### Epoch 机制

```
Epoch 是 Raft 中的 term 概念，每次主从切换 epoch+1：

EpochFile 记录：
┌─────────┬──────────────┬──────────────┐
│ Epoch   │ StartOffset  │ Timestamp    │
├─────────┼──────────────┼──────────────┤
│ 1       │ 0            │ T1           │
│ 2       │ 102400       │ T2           │
│ 3       │ 204800       │ T3           │
└─────────┴──────────────┴──────────────┘

作用：
- 确定主从切换的时间点
- 数据截断的依据
- 防止脑裂
```

### 角色切换

```java
// 切换为 Master
public boolean changeToMaster(int masterEpoch) {
    // ① 截断脏数据
    long truncateOffset = truncateInvalidMsg();

    // ② 更新 confirmOffset
    defaultMessageStore.setConfirmOffset(computeConfirmOffset());

    // ③ 追加新 Epoch
    EpochEntry newEpochEntry = new EpochEntry(masterEpoch, maxPhyOffset);
    epochCache.appendEntry(newEpochEntry);

    // ④ 等待 ConsumeQueue 分发完成
    while (defaultMessageStore.dispatchBehindBytes() > 0) {
        Thread.sleep(100);
    }

    // ⑤ 恢复 TopicQueueTable
    defaultMessageStore.recoverTopicQueueTable();
    return true;
}

// 切换为 Slave
public boolean changeToSlave(String newMasterAddr, int newMasterEpoch, Long slaveId) {
    // ① 停止所有连接
    destroyConnections();

    // ② 创建 HAClient 连接新 Master
    haClient = new AutoSwitchHAClient(...);
    haClient.updateMasterAddress(newMasterAddr);
    haClient.start();
    return true;
}
```

### SyncStateSet — 同步副本管理

```java
// SyncStateSet 记录当前同步的 Broker 集合
public class SyncStateSet {
    private Set<Long> syncStateSet;   // 同步副本 ID 集合
    private int masterEpoch;           // 当前 Master Epoch
}

// 当 Slave 追上 Master 时，加入 syncStateSet
// 当 Slave 断开时，从 syncStateSet 移除
// 只有在 syncStateSet 中的 Broker 才有资格被选为 Master
```

> [!tip] SyncStateSet 的作用
> 保证新选出的 Master **不会丢数据**：
> - 只有同步副本才有资格成为 Master
> - 异步副本可能落后太多，选为 Master 会丢数据

## 📐 3. Controller 模式 — 基于 DLedger 的选主

### 架构

```
┌──────────────────────────────────────────────────────────────┐
│                    Controller 模式架构                         │
│                                                              │
│  ┌─────────────────────────────────────────┐                 │
│  │ Controller 集群 (DLedger/Raft)           │                 │
│  │                                         │                 │
│  │ ┌──────────┐ ┌──────────┐ ┌──────────┐ │                 │
│  │ │Controller│ │Controller│ │Controller│ │                 │
│  │ │ Node-1   │ │ Node-2   │ │ Node-3   │ │                 │
│  │ │ (Leader) │ │(Follower)│ │(Follower)│ │                 │
│  │ └──────────┘ └──────────┘ └──────────┘ │                 │
│  │                                         │                 │
│  │ 管理：Broker 注册、选主、SyncStateSet    │                 │
│  └─────────────────────────────────────────┘                 │
│              ↑                    ↑                          │
│    ┌─────────┴──────┐  ┌─────────┴──────┐                   │
│    │ Broker Group A  │  │ Broker Group B  │                   │
│    │ ┌─────┐ ┌─────┐│  │ ┌─────┐ ┌─────┐│                   │
│    │ │ M   │ │ S   ││  │ │ M   │ │ S   ││                   │
│    │ └─────┘ └─────┘│  │ └─────┘ └─────┘│                   │
│    └────────────────┘  └────────────────┘                   │
└──────────────────────────────────────────────────────────────┘
```

### 核心接口

```java
public interface Controller {
    // Broker 注册
    CompletableFuture<RemotingCommand> registerBroker(RegisterBrokerToControllerRequestHeader request);

    // 申请 BrokerId
    CompletableFuture<RemotingCommand> applyBrokerId(ApplyBrokerIdRequestHeader request);

    // 选主
    CompletableFuture<RemotingCommand> electMaster(ElectMasterRequestHeader request);

    // 修改 SyncStateSet
    CompletableFuture<RemotingCommand> alterSyncStateSet(AlterSyncStateSetRequestHeader request, SyncStateSet syncStateSet);

    // 获取副本信息
    CompletableFuture<RemotingCommand> getReplicaInfo(GetReplicaInfoRequestHeader request);
}
```

### 选主流程

```
Master 宕机 → Broker 向 Controller 发送选主请求：

① Controller 收到 ElectMaster 请求
   │
   ▼
② ReplicasInfoManager.electMaster()
   │  使用 ElectPolicy 选出新 Master
   │  策略：优先选 syncStateSet 中的 Broker
   │
   ▼
③ Controller 返回新 Master 信息
   │  - 新 Master BrokerId
   │  - 新 Master 地址
   │  - 新 MasterEpoch
   │
   ▼
④ Broker 切换角色
   │  - 新 Master：changeToMaster()
   │  - 其他 Slave：changeToSlave(newMasterAddr)
   │
   ▼
⑤ 新 Master 向 Controller 注册
   │  - 更新 SyncStateSet
   │
   ▼
⑥ 故障转移完成 ✓
```

## 📐 三种 HA 模式对比

| 维度 | DefaultHA | AutoSwitchHA | Controller 模式 |
|------|-----------|-------------|----------------|
| 主从切换 | 手动 | 自动 | 自动 |
| 选主依据 | 配置 | SyncStateSet | DLedger Raft |
| 数据一致性 | 同步/异步复制 | SyncStateSet | Raft 共识 |
| 脑裂防护 | 无 | Epoch | DLedger Leader |
| 适用场景 | 4.x 简单部署 | 5.x 单集群 | 5.x 大规模部署 |
| 复杂度 | 低 | 中 | 高 |

## 📐 数据复制流程图

```
Producer 写入 → Master CommitLog
                    │
         ┌──────────┼──────────┐
         ▼          ▼          ▼
    Slave-1     Slave-2     Slave-3
    CommitLog   CommitLog   CommitLog

同步复制：
  Master 等待 Slave-1 写入成功 → 返回 Producer OK

异步复制：
  Master 立即返回 Producer OK
  Slave 异步拉取数据

Controller 模式：
  Master 等待 SyncStateSet 中的 Slave 写入成功
  → 返回 Producer OK
  → Controller 管理 SyncStateSet
```

## 💡 设计亮点

### 1. Epoch 防脑裂

```
每次主从切换 epoch+1，记录切换时间点和 offset
新 Master 启动时：截断 epoch 之后的脏数据
保证数据一致性
```

### 2. SyncStateSet 保证数据安全

```
只有同步副本才能被选为 Master
→ 新 Master 一定有最新数据
→ 不会丢消息
```

### 3. Controller 基于 Raft 共识

```
Controller 集群使用 DLedger（Raft）保证自身高可用
→ Controller Leader 宕机 → 自动选举新 Leader
→ 不影响 Broker 正常工作
```
