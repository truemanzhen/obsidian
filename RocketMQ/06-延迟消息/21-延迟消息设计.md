# 延迟消息 — 时间轮驱动的定时投递

> 📄 源码路径：`store/src/main/java/org/apache/rocketmq/store/timer/`
> 🏷️ #核心类 #设计模式 #面试高频

## 🎯 核心思想

延迟消息：Producer 发送消息时指定延迟时间，消息在指定时间后才对 Consumer 可见。

5.5.0 基于 **TimerWheel（时间轮）+ TimerLog（定时日志）** 实现，替代了 4.x 的固定延迟级别。

## 📐 整体架构

```
Producer
    │
    │ 发送延迟消息
    │ PROPERTY_TIMER_OUT_MS = deliverTimestamp
    ▼
┌──────────────────────────────────────────────────────────────┐
│ Broker                                                       │
│                                                              │
│  ① 消息写入 CommitLog（正常存储）                              │
│  ② 消息同时写入 %SYSTEM%wheel_timer 的 ConsumeQueue          │
│                                                              │
│  ┌──────────────────────────────────────────────────────┐    │
│  │ TimerMessageStore（定时消息引擎）                       │    │
│  │                                                      │    │
│  │  Enqueue 阶段（入时间轮）：                            │    │
│  │    TimerEnqueueGetService                            │    │
│  │    → 从 %SYSTEM%wheel_timer 读取消息                  │    │
│  │    → 计算延迟时间 → 写入 TimerLog + TimerWheel         │    │
│  │                                                      │    │
│  │  Dequeue 阶段（出时间轮）：                            │    │
│  │    TimerDequeueGetService                            │    │
│  │    → 扫描 TimerWheel 中到期的 Slot                    │    │
│  │    → TimerDequeuePutMessageService                   │    │
│  │    → 将消息重新写入原始 Topic 的 ConsumeQueue          │    │
│  │    → 对 Consumer 可见！                               │    │
│  └──────────────────────────────────────────────────────┘    │
└──────────────────────────────────────────────────────────────┘
```

## 📐 时间轮结构 — TimerWheel

```
TimerWheel 是一个环形数组，共 604800 个 Slot（7天 × 24小时 × 3600秒）：

精度：1 秒（可配置为 100ms）
Slot 数量：7 * 24 * 3600 = 604800

┌───┬───┬───┬───┬───┬───┬─────────────┬───┐
│ 0 │ 1 │ 2 │ 3 │ 4 │ 5 │ ... │ 604799│
└───┴───┴───┴───┴───┴───┴─────────────┴───┘
  ↑
  Slot[i] 存储：到期时间 = currWriteTimeMs + i * precisionMs

每个 Slot 指向 TimerLog 中的一条记录（链表结构）
```

> [!tip] 时间轮 vs 定时器
> | | Timer（ScheduledExecutorService） | TimerWheel（时间轮） |
> |---|---|---|
> | 数据结构 | 优先队列（堆） | 环形数组 |
> | 插入复杂度 | O(log n) | O(1) |
> | 触发复杂度 | O(log n) | O(1) |
> | 适用场景 | 少量定时任务 | 大量定时消息 |
>
> 时间轮是处理大量定时任务的经典数据结构（Netty HashedWheelTimer 也是这个原理）。

## 🔍 Enqueue 流程 — 消息入轮

```java
// TimerEnqueueGetService：从 ConsumeQueue 读取延迟消息
public boolean enqueue(int queueId) {
    ConsumeQueueInterface cq = this.messageStore.getConsumeQueue(TIMER_TOPIC, queueId);

    while (iterator.hasNext()) {
        CqUnit cqUnit = iterator.next();
        MessageExt msgExt = getMessageByCommitOffset(offsetPy, sizePy);

        // 读取延迟时间
        long delayedTime = Long.parseLong(msgExt.getProperty(TIMER_OUT_MS));

        // 创建 TimerRequest
        TimerRequest timerRequest = new TimerRequest(offsetPy, sizePy, delayedTime, ...);

        // 放入 enqueuePutQueue
        enqueuePutQueue.offer(timerRequest);
    }
}

// TimerEnqueuePutService：将消息写入 TimerWheel
public boolean doEnqueue(long offsetPy, int sizePy, long delayedTime, MessageExt messageExt) {
    // ① 计算是否需要 Roll（延迟时间超出当前窗口）
    boolean needRoll = delayedTime - currWriteTimeMs >= timerRollWindowSlots * precisionMs;

    // ② 写入 TimerLog（持久化）
    timerLog.appendTimerLog(offsetPy, sizePy, delayedTime, magic);

    // ③ 写入 TimerWheel（内存索引）
    timerWheel.putTimerRequest(timerLog.getWrotePosition() - TimerLog.UNIT_SIZE, delayedTime, needRoll);
}
```

## 🔍 Dequeue 流程 — 消息出轮

```java
// TimerDequeueGetService：扫描到期的 Slot
public boolean dequeue() {
    while (currReadTimeMs <= System.currentTimeMillis()) {
        // ① 从 TimerWheel 获取当前时间 Slot 中的记录
        ByteBuffer slotBuffer = timerWheel.getSlot(currReadTimeMs);

        // ② 遍历 Slot 中的所有 TimerLog 记录
        while (slotBuffer.hasRemaining()) {
            long timerOffset = slotBuffer.getLong();
            // 读取 TimerLog 记录
            TimerLog.TimerRecord record = timerLog.readTimerLog(timerOffset);

            // ③ 构建 TimerRequest，放入 dequeueGetQueue
            dequeueGetQueue.put(timerRequest);
        }

        // ④ 移动读指针
        currReadTimeMs += precisionMs;
    }
}

// TimerDequeuePutMessageService：将消息重新写入原始 Topic
public void doPutMessage(TimerRequest request) {
    // ① 读取原始消息
    MessageExt msgExt = getMessageByCommitOffset(request.getOffsetPy(), request.getSizePy());

    // ② 恢复原始 Topic（从 PROPERTY_REAL_TOPIC）
    msgExt.setTopic(msgExt.getProperty(MessageConst.PROPERTY_REAL_TOPIC));

    // ③ 删除延迟属性
    msgExt.removeProperty(TIMER_OUT_MS);
    msgExt.removeProperty(TIMER_ENQUEUE_MS);

    // ④ 重新写入 CommitLog（对 Consumer 可见！）
    PutMessageResult result = messageStore.putMessage(msgInner);
}
```

## 📐 延迟消息完整生命周期

```
时间线 ──────────────────────────────────────────────────────────→

① Producer 发送延迟消息（deliverTimestamp = T+300s）
   │
   ▼
② 消息写入 CommitLog + %SYSTEM%wheel_timer ConsumeQueue
   │
   ▼
③ TimerEnqueueGetService 读取消息
   │
   ▼
④ TimerEnqueuePutService 写入 TimerLog + TimerWheel
   │  TimerLog: 持久化记录 (offsetPy, sizePy, delayedTime)
   │  TimerWheel: Slot[T+300s] → 指向 TimerLog 记录
   │
   │  ... 等待 300 秒 ...
   │
   ▼
⑤ TimerDequeueGetService 扫描到期 Slot
   │  currReadTimeMs 到达 T+300s → 读取 Slot
   │
   ▼
⑥ TimerDequeuePutMessageService 重新写入
   │  恢复原始 Topic → 写入 CommitLog
   │
   ▼
⑦ ReputMessageService 分发到 ConsumeQueue
   │
   ▼
⑧ Consumer 消费到消息 ✓
```

## 💡 设计亮点

### 1. 精度可配置

```java
// 精度默认 1 秒，可配置为 100ms
this.precisionMs = storeConfig.getTimerPrecisionMs();  // 默认 1000ms

// Slot 数量固定为 7 天
this.slotsTotal = TIMER_WHEEL_TTL_DAY * DAY_SECS;  // 7 * 86400 = 604800
```

### 2. Roll 机制 — 处理超长延迟

```
当延迟时间超出当前时间窗口时：
  delayedTime - currWriteTimeMs >= timerRollWindowSlots * precisionMs

需要 Roll：将消息放到更远的 Slot，等时间轮转到后再处理
```

### 3. TimerLog — 持久化存储

```
TimerLog 是一个顺序写文件，存储每条延迟消息的元数据：
┌────────────┬────────────┬──────────────┬──────────┬──────────┐
│ offsetPy   │ sizePy     │ delayedTime  │ magic    │ prev     │
│ (8B)       │ (4B)       │ (8B)         │ (4B)     │ (8B)     │
└────────────┴────────────┴──────────────┴──────────┴──────────┘
总大小：32 字节

TimerWheel 内存中只存 TimerLog 的 offset（指针），不存完整数据。
```

### 4. 多线程 Dequeue

```java
// 多个 DequeueGet 线程并行读取
TimerDequeueGetMessageService[] dequeueGetMessageServices;

// 多个 DequeuePut 线程并行写入
TimerDequeuePutMessageService[] dequeuePutMessageServices;

// 通过 BlockingQueue 连接
dequeueGetQueue → dequeueGetMessageServices → dequeuePutQueue → dequeuePutMessageServices
```

## 📊 5.x vs 4.x 延迟消息对比

| 维度 | 4.x | 5.x |
|------|-----|-----|
| 实现 | 固定 18 个延迟级别 | TimerWheel 时间轮 |
| 精度 | 级别固定（1s/5s/10s/...） | 可配置（100ms/1s） |
| 灵活性 | 只能选级别 | 任意时间点 |
| 存储 | Broker 内部定时器 | TimerLog + TimerWheel |
| 性能 | 一般 | 高（时间轮 O(1)） |

## 📊 延迟属性

| 属性 | 含义 |
|------|------|
| `PROPERTY_TIMER_OUT_MS` | 目标投递时间（毫秒时间戳） |
| `PROPERTY_TIMER_ENQUEUE_MS` | 进入时间轮的时间 |
| `PROPERTY_TIMER_DEQUEUE_MS` | 离开时间轮的时间 |
| `PROPERTY_REAL_TOPIC` | 原始 Topic（时间轮用内部 Topic） |

## 💡 学习收获

1. **时间轮算法**：环形数组 + 定时扫描，O(1) 插入和触发
2. **TimerLog 持久化**：内存索引 + 磁盘日志，保证不丢消息
3. **Enqueue/Dequeue 分离**：入轮和出轮是独立的服务，互不影响
4. **Roll 机制**：处理超长延迟，消息可以跨越多个时间窗口
